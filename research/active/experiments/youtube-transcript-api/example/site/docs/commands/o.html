/o

Workflow Orchestration

The main command for all coding tasks. Plan, build, fix, explain, refactor, test, ship, and learn.

When to use this command

✓ Use /o when:

  • Creating new features (more than one file)
  • Fixing bugs across modules
  • Planning architecture
  • Explaining or learning code

× Don't use /o when:

  • Extracting design from images (use /d)
  • Full codebase review (use /r)

Basic Usage

/o [task description]

OCTOPUS auto-detects the best mode based on your request.

Special Commands

/o?

Deep init. Scans your entire project, loads learnings, activates all 9 minds. Use at the start of a session.

/o!

Emergency mode. For production issues. Triage → Stabilize → Fix. No refactors, no "while we're here" changes.

/o learn?

View current learnings (preferences, project patterns, corrections).

Modes

/o plan [task]

For complex tasks. Research → Plan → Execute. Shows your work before coding.

Triggers: New features, refactors >3 files, architecture decisions, DB/auth/payments

/o build [feature]

Build components and features. Production-ready code with no placeholders.

Output: Complete code, file locations, usage examples

/o fix [issue]

Debug and fix issues. Shows chain of thought: Read → Hypothesize → Verify → Fix.

Output: Problem, root cause, solution, prevention

/o explain [topic]

Get explanations adapted to your level. TL;DR, examples, key points.

/o refactor [target]

Safely restructure code. Tests pass before and after. Change structure, not behavior.

/o test [target]

Write tests. Covers happy path, edge cases, errors. Follows project's testing patterns.

/o ship

Deployment checklist. Quality, review, docs, deploy, rollback, monitor.

/o learn: [fact]

Teach OCTOPUS something. Saved to learnings for future sessions.

Auto-Switching

OCTOPUS automatically switches between Plan Mode (complex) and Agent Mode (simple) based on task complexity:

How /o chooses Plan vs Agent

  • Complex → Plan: Multi-file, architecture, new features
  • Simple → Agent: Single file, bug fixes, quick changes

Examples

/o plan user authentication with OAuth
/o build a Modal component with animations
/o fix the TypeError in UserService
/o explain how async/await works
/o refactor the API layer
/o test the checkout flow
/o learn: we use camelCase for variables
