## VENOCTIS Architecture — Core Decisions

### Heartbeat on VPS (not local)

**Why:** Always-on requires infrastructure that doesn't depend on Kariem's machine. VPS = independent life.

**Choice:** Hostinger KVM 4 (~$8/mo) — already have it, sufficient resources.

### MCP-first for senses

**Why:** Don't build scrapers from scratch. 50+ MCP servers exist. Fork, reshape, deploy.

**Key reshape targets:**
- `github/github-mcp-server` (official) → configure + webhook bridge
- `sparfenyuk/mcp-youtube` + `aihenryai/mcp-youtube-analytics` → merge best features
- `jjsymes/mcp-rss-reader` → add Arabic sources + relevance scoring

### Webhook > Polling

**Why:** Polling = 15min latency. Webhooks = instant. For code review, instant matters.

**Architecture:** nginx SSL reverse proxy → webhook_server.py → spawns code_reviewer.py worker → Telegram alert.

### Telegram as voice

**Why:** Two-way. Mobile-native. Already in OpenClaw. Kariem checks it constantly.

**Not Discord/Slack:** Those are team tools. Telegram is personal. VENOCTIS is Kariem's personal intelligence layer.

### Self-tasking from Phase 3

**Why:** VENOCTIS creates its own work (Dependabot alerts → create task → fix → PR). Autonomy = not waiting for commands.

**Approval workflow:** Some tasks auto-allowed (dependency patch, INDEX rebuild). Some need approval (architecture changes). Priority queue (P0-P3).

### Git as memory bridge

**Why:** Claude.ai project can't directly write to VPS. VPS can't directly write to Claude.ai. Git is the neutral bridge.

**Flow:** VENOCTIS updates `.venom/MEMORY.md` → pushes to repo → Claude.ai project sees it on next session. Bi-directional.
